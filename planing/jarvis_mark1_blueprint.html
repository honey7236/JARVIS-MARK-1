JARVIS-MARK-1 full project blueprint — file structure, module roles, execution flow, and dependencies

Entry points
JARVIS.py — CLI / voice loop. The main brain. Boots up speech, memory, and the command processor.
app.py — GUI launcher. Uses eel + Edge browser to show the frontend, then runs JARVIS in a background thread.
Core layers
Brain/ — AI command processor (LLM integration)
core/ — STT (listen) + TTS (speak)
automation/ — system tasks, weather, news, internet
memory/ — short-term session + long-term learning
frontend/ — HTML/CSS/JS dashboard UI
data/ — persistent JSON stores
How it works — one-liner
JARVIS listens via microphone → transcribes speech with speech_recognition → feeds command into Brain.py (LLM / rule engine) → runs automation if needed → stores to memory → speaks response via pyttsx3. The GUI mirrors all of this via eel WebSockets.
Language breakdown
Python 91% Core engine, all modules
JS 5% Frontend interactivity (eel bridge)
CSS 2.3% Dashboard styling
HTML 1.4% index.html shell
Key design decisions
GUI runs JARVIS in a daemon thread — crash-safe, non-blocking
Short-term memory is a session list saved to JSON on each turn
Long-term learning updates a profile from command patterns
All modules added to sys.path via relative path injection

Brain/Brain.py
Role: Central command router. Receives a raw text command and returns a response string.

Key function: processcomand(command)

Likely integrates an LLM API (Gemini / OpenAI) or a rule-based intent classifier. Returns plain text that JARVIS will speak aloud.
core/speech_to_text.py
Role: Microphone input → text.

Key function: listen() → str | None

Uses speech_recognition (Google STT). Returns None on silence/error so the main loop can safely skip.
core/text_to_speech.py
Role: Text → spoken audio output.

Key function: speak(text)

Uses pyttsx3 (offline TTS engine). Configurable voice, rate, and volume. Blocks until audio finishes.
automation/datetimegreet.py
Role: Time-aware greeting at startup.

Key function: greet_user() → str

Returns "Good morning / afternoon / evening, sir" based on current system time.
automation/weather.py
Role: Fetch current weather data.

Key function: get_weather() → dict

Calls an external weather API (likely OpenWeatherMap). Result is displayed in the GUI dashboard and can be spoken on command.
automation/news.py
Role: Fetch latest headlines.

Key function: get_news() → list[str]

Uses a news API or RSS feed. Headlines are shown in the frontend and can be read aloud.
automation/system_info.py
Role: System resource monitoring.

Key function: get_system_stats() → dict

Returns CPU%, RAM%, disk usage via psutil. Feeds the live stats panel in the dashboard.
automation/internet_check.py
Role: Network connectivity status.

Key function: internet_status() → dict

Pings a known host and returns connection state + latency. Shown in the dashboard header.
memory/short_term.py
Role: Session conversation log.

Key functions: add_message(role, text), save_session(), load_session()

Keeps a list of {role, text} dicts in memory. Persists to data/session.json after every turn.
memory/learning.py
Role: Long-term user profiling.

Key function: learn_user(command)

Parses commands for patterns (frequent topics, preferred name, common tasks) and updates data/user_profile.json.
frontend/index.html + JS
Role: Live dashboard rendered by Microsoft Edge via eel.

Shows: internet status, weather, news, system stats. The eel WebSocket bridge lets Python push data to JS in real time and vice versa.
CLI mode — JARVIS.py
1
Boot — speak("initializing JARVIS…") confirms TTS is alive
2
Greet — greet_user() generates time-aware greeting, spoken aloud
3
Load session — load_session() restores last conversation from data/session.json
4
Listen — listen() opens mic, waits for speech, returns text. Loops back if None.
5
Log input — add_message("you", command) appends to session
6
Learn — learn_user(command) updates long-term profile if patterns detected
7
Process — processcomand(command) in Brain.py routes to LLM or automation handler, returns response string
8
Log output — add_message("jarvis", response) + save_session()
9
Exit check — if response == "exit", speak("Shutting down") and break. Otherwise speak the response and loop to step 4.
GUI mode — app.py
1
Path setup — resource_path() resolves frontend folder for both dev and PyInstaller .exe
2
eel init — eel.init("frontend") sets the web root
3
Thread spawn — threading.Thread(target=jarvis, daemon=True).start() runs the full voice loop in the background
4
Dashboard data — internet_status(), get_weather(), get_news(), get_system_stats() called to pre-populate the UI
5
Launch GUI — eel.start("index.html", mode="edge", host="localhost", port=8000) opens the Edge browser window. This call blocks until the window is closed.
Data flow — memory lifecycle
Each conversation turn follows this path through memory:
listen() → raw text → add_message("you", …) → learn_user(…) → processcomand(…) → add_message("jarvis", …) → save_session()

Short-term (session.json) is reloaded at startup. Long-term (user_profile.json) accumulates across sessions and can inform Brain.py's responses.
